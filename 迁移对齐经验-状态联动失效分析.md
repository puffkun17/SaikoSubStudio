# 迁移对齐经验总结：状态联动失效分析

## 问题现象（多次出现）

- Timeline 进度条 / 行号输入可以正常拖动和跳转
- `previewIndex` 状态也能被更新
- 但字幕列表（SequenceList）完全不动：不滚动、不高亮、不响应
- Theater 区域也可能无法正确显示当前字幕
- 上一次迁移对齐后也出现过几乎完全相同的症状

## NAS 版本的核心设计

NAS 版本采用的是**明确的职责分离 + 响应式驱动**模式：

### 1. TimelineControls 的职责
- 仅负责更新两个状态：
  - `setPreviewIndex(index)`
  - `setJumpLineVal(line)`
- **不直接操作 DOM**（不直接 scroll）
- 在 `onMouseUp` 时调用 `handleJumpToLine()`，里面只是触发滚动提示

### 2. SequenceList 的职责（关键）
- 通过 `useEffect` 监听 `previewIndex` 的变化
- 内部维护 `prevIndexRef` 避免重复触发
- 使用 `setTimeout` + `scrollTo` 实现平滑居中滚动
- 滚动逻辑完全封装在列表组件内部

### 3. 状态流动
```
TimelineControls → setPreviewIndex → Store → SequenceList useEffect → 滚动 + 高亮
```

NAS 版本把“滚动”这件事彻底委托给了列表组件自己，而不是 Timeline 去指挥 DOM。

## Git 版本的问题根源

虽然 `TimelineControls.tsx` 和 `SequenceList.tsx` 两个文件**代码几乎一模一样**，但联动依然失效，说明问题出在**更上游的状态连接层**。

### 可能原因（按概率排序）

1. **processedSubs 与 previewIndex 的更新不同步（最高概率）**
   - Timeline 在 `onChange` 时直接调用 `setPreviewIndex`
   - 但如果 `processedSubs` 是空数组、旧引用、或尚未就绪，`SequenceList` 的渲染条件或 `useEffect` 依赖可能不满足，导致不触发滚动。

2. **Store 层面状态定义不完整**
   - Git 版本的 `useStudioStore.ts` 缺少部分 NAS 版本中与时间轴联动相关的状态或 action（例如 `alignmentMode`、`refScreenshot` 等字段虽然不直接相关，但反映出 store 可能被裁剪过）。
   - `previewIndex` 的初始化值或更新方式可能存在差异。

3. **父组件传递链断裂**
   - IngestStep / Workbench 等中间层组件在之前的迁移中被简化，导致 store 的订阅关系没有正确建立。

4. **多次迁移的系统性问题**
   - 之前的提取过程可能只保留了“视觉和局部逻辑”，而丢失了**状态驱动的响应式连接**（reactive connection）。
   - 这解释了为什么上一次迁移也出现同样问题：问题不在单个组件，而在“状态流动”这个更抽象的层面。

## 经验教训

1. **不要只对齐单个组件的外观和局部逻辑**
   - 必须把**状态流动路径**（data flow）作为第一优先级对齐对象。

2. **关键状态的响应式连接最容易丢失**
   - `previewIndex` + `processedSubs` + `jumpLineVal` 这三者的联动关系，必须从 store 层面开始验证。

3. **滚动逻辑的委托模式很重要**
   - NAS 把滚动委托给列表组件的 useEffect，这是正确的模式。Git 版本虽然复制了代码，但如果上游状态不稳定，这个模式就会失效。

4. **迁移时建议的检查清单**
   - [ ] Store 中与时间轴相关的所有状态是否完整
   - [ ] `previewIndex` 的更新是否能触发下游组件的 useEffect
   - [ ] `processedSubs` 的生成时机是否早于 Timeline 操作
   - [ ] 是否存在“先更新 previewIndex，后 processedSubs 才就绪”的竞态条件
